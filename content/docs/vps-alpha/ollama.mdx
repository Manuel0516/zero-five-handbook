Ollama runs **natively on the host** (not in Docker) so every project on the server can share one model runtime. Version at time of writing: `0.30.8`, installed at `/usr/local/bin/ollama` via the official installer.

The defining configuration choice: **Ollama binds to the VPN address, not localhost and not the public interface.**

```text
OLLAMA_HOST=10.8.0.1:11434
```

This makes it reachable from the [WireGuard](/docs/vps-alpha/wireguard) network (the laptop) and from Docker containers on the host, while being completely invisible to the internet.

---

## systemd service

`/etc/systemd/system/ollama.service` (created by the installer, runs as the `ollama` user) plus a drop-in override at `/etc/systemd/system/ollama.service.d/override.conf`:

```ini
[Service]
Environment="OLLAMA_KEEP_ALIVE=10m"
Environment="OLLAMA_HOST=10.8.0.1:11434"
```

| Setting | Value | Why |
|---------|-------|-----|
| `OLLAMA_HOST` | `10.8.0.1:11434` | VPN-only binding — never public |
| `OLLAMA_KEEP_ALIVE` | `10m` | Models unload after 10 minutes idle; the box only has 8 GB RAM |

Edit the override with `sudo systemctl edit ollama`, then `sudo systemctl restart ollama`.

## Models

Verified with `ollama list`:

| Model | Size | Used by |
|-------|------|---------|
| `qwen2.5:3b` | 1.9 GB | General chat via [Open WebUI](/docs/vps-alpha/ai-stack) |
| `translategemma:4b` | 3.3 GB | [Telegram translation bot](/docs/vps-alpha/telegram-bot) |
| `nomic-embed-text` | 274 MB | Embeddings (Open WebUI RAG) |

Heavier models (`deepseek-r1:7b`, `qwen2.5-coder:7b`) intentionally live on the **laptop's** Ollama (`10.8.0.2:11434`) and are reached over the VPN — the VPS doesn't have the RAM for them.

## CLI usage

Because Ollama isn't on the default `127.0.0.1:11434`, the CLI needs `OLLAMA_HOST` set (the `manuel` user is in the `ollama` group):

```bash
export OLLAMA_HOST=10.8.0.1:11434   # add to ~/.bashrc for convenience

ollama list          # installed models
ollama ps            # currently loaded models
ollama pull qwen2.5:3b
ollama run qwen2.5:3b "hello"
ollama stop qwen2.5:3b   # force-unload before the 10m keep-alive
```

Or query the API directly:

```bash
curl http://10.8.0.1:11434/api/tags
```

## Setup from scratch

<Steps>

### Bring up WireGuard first

Ollama binds to `10.8.0.1`, which only exists once [wg0 is up](/docs/vps-alpha/wireguard). Otherwise the service fails to bind.

### Install with the official script

```bash
curl -fsSL https://ollama.com/install.sh -o /tmp/ollama-install.sh
less /tmp/ollama-install.sh     # review before running
sh /tmp/ollama-install.sh
```

The installer creates the `ollama` user and the systemd unit. Add yourself to the group: `sudo usermod -aG ollama manuel`.

### Configure the override

```bash
sudo systemctl edit ollama
```

```ini
[Service]
Environment="OLLAMA_KEEP_ALIVE=10m"
Environment="OLLAMA_HOST=10.8.0.1:11434"
```

```bash
sudo systemctl daemon-reload && sudo systemctl restart ollama
```

### Pull the models

```bash
export OLLAMA_HOST=10.8.0.1:11434
ollama pull qwen2.5:3b
ollama pull translategemma:4b
ollama pull nomic-embed-text
```

### Verify

```bash
curl http://10.8.0.1:11434/api/tags
systemctl status ollama
```

</Steps>

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| `ollama list` says connection refused | `OLLAMA_HOST` not exported — the CLI is trying `127.0.0.1` |
| Service fails at boot | wg0 not up yet — check `systemctl status wg-quick@wg0` first |
| Out of memory | Check `ollama ps` for loaded models; lower `OLLAMA_KEEP_ALIVE`; remember the Minecraft server competes for the same 8 GB |
