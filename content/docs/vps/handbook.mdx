Every push to `master` of [zero-five-handbook](https://github.com/Manuel0516/zero-five-handbook) is designed to validate, rebuild, and update `https://docs.zero-five.space`. Pull requests run the same validation but cannot deploy. The site is intended to be reachable **only over WireGuard**.

---

## Why a server and not static files

The site is Next.js + Fumadocs with two dynamic API routes — `/api/chat` (AI search via OpenRouter) and `/api/search` — plus a request-time proxy for Markdown content negotiation. A static export would drop those, so it runs as a Node server in a container (`next build` with `output: 'standalone'`).

---

## The VPN-only pattern

Two layers make the site private:

1. **DNS points into the tunnel.** `docs.zero-five.space` has an explicit A record → `10.8.0.1` (grey cloud), overriding the wildcard that points at the public IP. Every WireGuard peer routes `10.8.0.1/32` through `wg0`, so on the VPN the name just works; off the VPN it resolves to an unreachable private address.
2. **Traefik rejects non-VPN sources.** Anyone can still connect to the *public* IP and send `Host: docs.zero-five.space`, so the router carries an `ipWhiteList` middleware allowing only `10.8.0.0/24`. Docker's NAT preserves source addresses, so requests arriving through `wg0` show their real `10.8.0.x` origin and everything else gets a 403.

HTTPS still works because certificates come from the **DNS-01 challenge** (see [Traefik](/docs/vps/traefik)) — Let's Encrypt never connects to the address the record points at.

> **Note:** No firewall change was needed: Traefik's published `:443` bypasses UFW (Docker trap #1 on the [Security](/docs/vps/security) page), which here works in our favor — the port is already reachable on `wg0`.

## The pipeline

`.github/workflows/deploy.yml` is triggered for pull requests and pushes to `master`, and can also be started manually. A deployment happens only after locked dependency installation, type checking, linting, the Next.js production build, and a production Docker image build all succeed.

1. GitHub Actions **SSHes into the VPS** using a dedicated deploy key and a pinned host key.
2. The VPS runs **`git pull --ff-only`** in `~/services/zero-five-handbook/`, which is a clone of this repo (the VPS's own read-only deploy key authenticates to GitHub).
3. Compose validates its resolved model, rebuilds the image, recreates the container, and waits for its `/docs` health check.
4. The job proves that Docker published no handbook host port, checks `/docs` inside the container, and checks HTTPS through Traefik at `10.8.0.1` before pruning dangling images.

| Piece | Location |
|-------|----------|
| Dockerfile, compose file, workflow | handbook repo root / `.github/workflows/` |
| Stack on the VPS | `~/services/zero-five-handbook/` — a git clone of the repo |
| `OPENROUTER_API_KEY` | `~/services/zero-five-handbook/.env` (gitignored, never in git) |
| Deploy SSH key | Private half in repo secret `VPS_SSH_KEY`; public half in `~/.ssh/authorized_keys` on the VPS |
| VPS endpoint | Repo secret `VPS_HOST`; use the public IP or a publicly resolvable management hostname, without a port |
| VPS host identity | Verified complete `known_hosts` line in repo secret `VPS_SSH_KNOWN_HOSTS` |

> **Warning:** The Next.js build is the heaviest thing this box runs (the 8 GB of RAM have no swap behind them). If a deploy fails with a killed build process, check `free -h` and stop the Minecraft server before retrying.

The runtime image is Next's **standalone** output: `server.js` plus traced `node_modules` and the static assets, on `node:24-alpine`, listening on `:3000` inside the `traefik` network — never published on the host.

The `.env` file is optional. Without it the handbook and search route still
run, but AI chat has no OpenRouter credential. To enable chat, create the
root-readable VPS file with `OPENROUTER_API_KEY` and optionally
`OPENROUTER_MODEL`; Compose substitutes those values at runtime and they never
enter the image.

## One-time GitHub and VPS setup

Create a dedicated SSH key for Actions, add only its public key to the VPS
user's `~/.ssh/authorized_keys`, and put the private key in `VPS_SSH_KEY`. The
key should not be the VPS key that reads GitHub; these are two separate trust
directions.

Pin the server identity instead of trusting a fresh `ssh-keyscan` result during
every deployment. From an already trusted connection, inspect the VPS SSH host
fingerprint and compare it with the server's host public key. Then save the
complete bracketed-port line as `VPS_SSH_KNOWN_HOSTS`, for example:

```text
[vps-address]:2020 ssh-ed25519 AAAA...verified-host-public-key...
```

Create a GitHub environment named `production` if deployment approvals or
environment-scoped secrets are wanted. Repository secrets also work with the
workflow as written.

## Rollback

Deploys are stateless (no volumes, no migrations), so rollback is git:

```bash
git revert <bad commit> && git push   # pipeline redeploys the previous state
```

Or re-run the workflow from an older commit in the Actions tab (`workflow_dispatch` is enabled).

If a container starts but fails its health check, inspect it before changing
networking:

```bash
cd ~/services/zero-five-handbook
docker compose ps
docker compose logs --tail=200 handbook
git log -2 --oneline
```

Revert the bad source commit and let the workflow rebuild. Do not stop Traefik,
remove its external network, or alter WireGuard as part of an application
rollback.

## Troubleshooting

| Symptom | Check |
|---------|-------|
| 403 from Traefik | You are not on WireGuard, or the request reached the public IP — `dig docs.zero-five.space` should return `10.8.0.1` |
| Name doesn't resolve to `10.8.0.1` | The explicit A record is missing/edited in Cloudflare; the wildcard would take over |
| Site unreachable on VPN | `ping 10.8.0.1` first (tunnel up?), then `docker ps` on the VPS for the `handbook` container |
| AI search fails | `docker logs handbook` — usually a missing/invalid `OPENROUTER_API_KEY` in `~/services/zero-five-handbook/.env` |
| Deploy job fails at SSH | Key mismatch: repo secret `VPS_SSH_KEY` vs `~/.ssh/authorized_keys` on the VPS |
| `Host key verification failed` | `VPS_SSH_KNOWN_HOSTS` does not contain the verified `[host]:2020` host-key line, or the VPS host key genuinely changed |
| `docker compose ... --wait` fails | Run `docker compose ps` and `docker compose logs --tail=200 handbook`; the app health check did not become healthy within three minutes |
| HTTPS verification returns 403 | The VPS-local request did not arrive with a `10.8.0.0/24` source; inspect Traefik access logs before changing the allowlist |

The workflow proves the allowed path from the VPS, but it cannot prove the
outside-VPN denial from a GitHub-hosted runner because public DNS intentionally
returns the private VPN address. Complete that boundary test from a device not
connected to WireGuard, forcing the public VPS address while preserving the
hostname:

```bash
curl --head --resolve docs.zero-five.space:443:PUBLIC_VPS_IP \
  https://docs.zero-five.space/
```

Expected result: Traefik returns `403`, and its access log shows that the
request never reached the handbook container.
