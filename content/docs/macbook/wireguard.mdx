The MacBook joins the [VPS-centered WireGuard network](/docs/vps-alpha/wireguard) as
`10.8.0.3/32`. It can reach services on `10.8.0.1`, while ordinary internet
traffic and the CachyOS laptop remain outside its WireGuard route.

## Current state

| Item | State |
| --- | --- |
| Installation | Homebrew `wireguard-tools` with userspace `wireguard-go` |
| Commands | `/opt/homebrew/bin/wg` and `/opt/homebrew/bin/wg-quick` |
| VPN address | `10.8.0.3/32` |
| Deployed config | `/opt/homebrew/etc/wireguard/zero-five.conf`, mode `0600` |
| Runtime | VPS ping and VPN-bound Ollama verified |
| VPS peer | Runtime and `/etc/wireguard/wg0.conf` persistence verified |
| Mac startup | Activation after reboot or login remains unverified |

The App Store WireGuard application was installed initially but is not used for
this setup. The command-line workflow makes the configuration and inspection
steps explicit.

## Install and inspect the CLI

```bash
brew install wireguard-tools
command -v wg
command -v wg-quick
wg --version
brew list --versions wireguard-tools wireguard-go
```

Installing the tools does not create keys, routes, or a tunnel. On macOS,
`wireguard-go` creates a userspace tunnel attached to a dynamic `utunN`
interface. The number may change and must never be hard-coded.

## Routing policy

```text
MacBook                              VPS
10.8.0.3/32  -------------------->  10.8.0.1/24
AllowedIPs: 10.8.0.1/32             UDP endpoint :51820

              no WireGuard route to 10.8.0.2
```

Because the peer's `AllowedIPs` contains only `10.8.0.1/32`, macOS installs one
host route to the VPS. It does not replace the default route. The VPS firewall
also denies routed traffic between WireGuard clients.

## Files and ownership

| Location | Path | Purpose |
| --- | --- | --- |
| Infrastructure repo | `wireguard/macbook.conf.example` | Secret-free operational source |
| MacBook | `~/.config/wireguard/keys/macbook.key` | Private key, mode `0600` |
| MacBook | `~/.config/wireguard/keys/macbook.pub` | Public key |
| MacBook | `/opt/homebrew/etc/wireguard/zero-five.conf` | Deployed tunnel config, mode `0600` |
| VPS | `/etc/wireguard/wg0.conf` | Persistent hub config, root-owned and mode `0600` |

The example contains the real public routing metadata and a
`MACBOOK_PRIVATE_KEY` placeholder. There is no generator. The private value is
inserted only into the Mac's root-readable deployed copy.

## Create or rebuild the Mac configuration

Generate a keypair only if the Mac does not already have one:

```bash
umask 077
mkdir -p ~/.config/wireguard/keys
wg genkey | tee ~/.config/wireguard/keys/macbook.key \
  | wg pubkey > ~/.config/wireguard/keys/macbook.pub
chmod 600 ~/.config/wireguard/keys/macbook.key \
  ~/.config/wireguard/keys/macbook.pub
```

Never regenerate an active keypair casually: doing so changes the public
identity expected by the VPS.

Validate and copy the source:

```bash
cd /Users/manuel/Desktop/zero-five/zero-five-infra
make validate
sudo install -m 600 wireguard/macbook.conf.example \
  /opt/homebrew/etc/wireguard/zero-five.conf
sudoedit /opt/homebrew/etc/wireguard/zero-five.conf
```

In `sudoedit`, replace `MACBOOK_PRIVATE_KEY` with the existing private key.
Review the file without pasting it into chat or logs. Confirm permissions:

```bash
sudo stat -f '%Sp %Su:%Sg %N' \
  /opt/homebrew/etc/wireguard/zero-five.conf
```

Expected mode is `-rw-------`. A failed edit can be rolled back by stopping the
tunnel and restoring the previous root-owned copy.

## Add the Mac peer safely

This procedure is already complete for the current Mac identity. Use it only
for recovery or an intentional identity replacement.

1. Run `make validate` locally.
2. Keep the existing VPS SSH session open and establish a second session.
3. On the VPS, back up `/etc/wireguard/wg0.conf` with root ownership and mode
   `0600`.
4. Add the Mac public key to the running interface first:

```bash
sudo wg set wg0 peer MACBOOK_PUBLIC_KEY allowed-ips 10.8.0.3/32
```

5. Start the Mac tunnel and validate both machines.
6. Only after runtime validation, add the same public peer block to the VPS
   persistent config. Do not overwrite its private key or laptop peer.

The runtime-first sequence protects the existing tunnel. `wg set` changes the
kernel's live WireGuard device without restarting `wg0`; a bad new peer can be
removed without disrupting the established laptop peer.

## Start and validate

```bash
sudo wg-quick up zero-five
sudo wg show
route -n get 10.8.0.1
route -n get 10.8.0.2
route -n get default
ping -c 3 10.8.0.1
curl --fail --show-error http://10.8.0.1:11434/api/tags
```

Expected results:

- `10.8.0.1` uses a dynamic `utunN` interface;
- `10.8.0.2` does not use the WireGuard tunnel;
- the default route remains on the normal network interface;
- the VPS responds to ping and the private Ollama API;
- `sudo wg show` reports a recent handshake.

On the VPS, also confirm the laptop peer, UFW, and `wg-quick@wg0` remain in their
previous state. A handshake alone does not prove route or application policy.

## Common failures

| Symptom | Check |
| --- | --- |
| `wg-quick` rejects the config | Private-key placeholder was not replaced, syntax, or permissions |
| No handshake | VPS endpoint, UDP firewall, public-key match, internet access |
| Handshake but no VPS access | `AllowedIPs`, macOS route, VPS UFW, service binding |
| Internet unexpectedly uses VPN | `AllowedIPs` is broader than `10.8.0.1/32` |
| Works manually but not after reboot | No verified startup service yet |

## Rollback

Stop the Mac tunnel:

```bash
sudo wg-quick down zero-five
```

If this was a new or replacement peer, remove only that live peer on the VPS:

```bash
sudo wg set wg0 peer MACBOOK_PUBLIC_KEY remove
```

If persistent configuration changed, restore the protected VPS backup and
restart `wg-quick@wg0` from the still-open SSH session. Recheck the laptop peer
and private Ollama path. Do not change UFW or SSH during this rollback.

To remove only the Mac CLI after the tunnel is down:

```bash
brew uninstall wireguard-tools
```

Review `brew autoremove` separately before allowing it to remove dependencies.
