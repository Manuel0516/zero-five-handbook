WireGuard creates the private `10.8.0.0/24` network centered on the VPS. The
network carries traffic between the VPS and CachyOS laptop so the
two [Ollama](/docs/vps/ollama) instances can communicate **without being exposed
to the public internet**. The MacBook is also active with VPS-only access.

Exact inventory and architecture decisions now live in `zero-five-infra`.
Legacy sanitized templates remain in the
[personal-ai repo](https://github.com/Manuel0516/personal-ai) under
`wireguard/` until the new source configuration replaces them.

## Topology

```text
                                 VPS hub
                          wg0: 10.8.0.1/24
                          ListenPort 51820/udp
                            /              \
                    active /                \ active
                          /                  \
       CachyOS laptop                         MacBook
       wg0: 10.8.0.2/24                       10.8.0.3/32
       PersistentKeepalive 25                 VPS-only access
       Ollama: 10.8.0.2:11434                 runtime verified
```

| Machine | VPN IP | Role |
|---------|--------|------|
| VPS | `10.8.0.1` | Server — listens on UDP `51820` (UFW-allowed) |
| Laptop | `10.8.0.2` | Peer — dials out, keeps the tunnel alive from behind NAT |
| MacBook | `10.8.0.3` | Active peer — reaches VPS services only |

The interface is managed by systemd: `wg-quick@wg0.service` is **enabled** and
active on the VPS. `ollama-router` uses Docker host networking, so it shares the
VPS network namespace and reaches both Ollama backends through the host's
WireGuard route.

The VPS is the hub, but it does not currently route traffic between clients:
kernel IPv4 forwarding is enabled while UFW's default routed policy is `deny`.
The MacBook therefore cannot reach the laptop through WireGuard unless a later
policy change explicitly permits it.

Runtime Mac connectivity is verified. The Mac can ping `10.8.0.1` and query the
VPN-bound Ollama API, while `10.8.0.2` and the ordinary default route remain on
the Mac's normal network interface. The owner also verified that the Mac peer
is present in the persistent `/etc/wireguard/wg0.conf` configuration. Automatic
activation after a Mac reboot or login remains unverified.

## Configuration

The real config is at `/etc/wireguard/wg0.conf` (root-only, contains the private key). Sanitized templates are committed in the repo:

### Server — `wireguard/server/wg0.conf.example`

```ini
[Interface]
Address = 10.8.0.1/24
ListenPort = 51820
PrivateKey = SERVER_PRIVATE_KEY

[Peer]
# Laptop
PublicKey = LAPTOP_PUBLIC_KEY
AllowedIPs = 10.8.0.2/32
```

### Laptop — `wireguard/laptop/wg0.conf.example`

```ini
[Interface]
Address = 10.8.0.2/24
PrivateKey = LAPTOP_PRIVATE_KEY

[Peer]
# Server
PublicKey = SERVER_PUBLIC_KEY
Endpoint = SERVER_PUBLIC_IP:51820
AllowedIPs = 10.8.0.1/32
PersistentKeepalive = 25
```

`PersistentKeepalive = 25` keeps the NAT mapping open so the VPS can reach the laptop even when the laptop initiated the tunnel.

> **Warning:** `AllowedIPs` is deliberately narrow (`/32` per peer) — the tunnel only routes the two VPN addresses, nothing else. This is a point-to-point link, not a full-tunnel VPN.

## Setup from scratch

> **Source of truth:** the existing VPS–laptop examples below remain useful for
> the original link. Mac inventory, peer ownership, secret-free templates, and
> the complete runtime-first deployment and rollback procedure live in
> `zero-five-infra`. The Mac private key remains only on the MacBook.

<Steps>

### Install WireGuard on both machines

```bash
sudo apt install wireguard        # VPS (Ubuntu)
sudo pacman -S wireguard-tools    # laptop (CachyOS — official repo, no AUR)
```

### Generate a keypair on each machine

```bash
umask 077
wg genkey | tee private.key | wg pubkey > public.key
```

Never commit or copy private keys between machines — only exchange the **public** keys.

### Write the configs

Copy the matching template to `/etc/wireguard/wg0.conf` on each machine and fill in keys, the peer's public key, and (on the laptop) the VPS IP `95.111.229.142`.

### Open the firewall on the VPS

```bash
sudo ufw allow 51820/udp
```

### Enable on both sides

```bash
sudo systemctl enable --now wg-quick@wg0
```

</Steps>

## Verifying the tunnel

```bash
sudo wg                              # handshake age, transfer counters
ip addr show wg0                     # interface is up with 10.8.0.1/24
ping 10.8.0.2                        # laptop reachable
curl http://10.8.0.2:11434/api/tags  # laptop Ollama reachable (only when laptop is on)
```

A recent `latest handshake` in `sudo wg` output means the tunnel is healthy. No handshake usually means the laptop is off — that is normal; the [ollama-router](/docs/vps/ai-stack#ollama-router) handles laptop absence gracefully.

## Security rules

From the repo's `docs/security.md` — the operating principles:

1. Never expose Ollama publicly; it binds to VPN IPs only.
2. Only UDP `51820` is open on the VPS for WireGuard.
3. The laptop's Ollama binds to `10.8.0.2:11434`, never to LAN or public interfaces.
4. Private keys never leave their machine and are never committed.
5. New peers know only the VPS by default; client-to-client access requires an explicit policy decision.
