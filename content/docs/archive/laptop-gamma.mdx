## Preserved artifacts

The recovery scope deliberately contains only the requested services. The
planned transfer destination is a USB drive prepared directly on Gamma.

| Archive | Source | Contents | Intentional exclusions |
| --- | --- | --- | --- |
| Minecraft | `~/MinecraftServer` | The active Java server, world, mods, configuration, and logs | None from the server folder |
| TelegramBot code | `~/TelegramBot` | Current source and project metadata | `node_modules/` and `.env` |
| Hall | `~/Hall` | Vite source and built `dist/` site | `node_modules/` |

The Minecraft capture was made while the service was running. It is a valuable
preservation copy, but not a filesystem-consistent backup. Before relying on it
for a future server, validate it after extraction and take a clean replacement
copy with the server stopped or after an in-game save.

The TelegramBot `.env` was intentionally not copied because it may contain a
Telegram token. Recreate the configuration on the replacement system with a
new token rather than reusing the old secret.

An attempted network transfer to the MacBook was cancelled because the LAN
connection repeatedly dropped long transfers. All temporary archives, chunks,
and copied files from that attempt were removed. The USB transfer is not yet
complete; do not mark any preservation copy as complete until its SHA-256
checksums have been verified on the USB.

commits and untracked project files. Neither was included in the requested
archive.

## Security incident

Gamma was compromised at inspection. A user-owned executable impersonating
`systemd-resolved` ran from a hidden directory in the home folder. It was a
static 64-bit ELF binary, launched twice by cron, and consumed most of the
CPU. Additional cron entries attempted to launch transient files from `/tmp`
and the user runtime directory.

The identified malicious processes were terminated and the user crontab,
binary, associated configuration, and transient watcher paths were removed.
That containment action does **not** establish that the machine is clean: it
could contain other persistence or stolen credentials. The correct recovery is
an OS reinstall, not continued operation.

### Recovery rules

1. Keep Gamma off untrusted networks until it is wiped.
2. Rotate the laptop SSH password, dynamic-DNS credentials, Telegram bot token,
   and any credentials stored in the old home directory from a clean device.
3. Treat the saved archives as inert data: do not execute scripts or restore
   dependencies blindly. Scan them on the replacement system before extraction.
4. Rebuild services from reviewed source, current dependencies, and newly
   created secrets. Do not restore the old PM2 state, cron entries, or hidden
   user-system directories.

## Restore checks

After transfer, verify each archive against its recorded SHA-256 checksum and
list its entries before extracting. Restore Minecraft in an isolated folder
first; restore TelegramBot and Hall as source projects, reinstall dependencies
from their manifest files, and provide new runtime configuration manually.

The next operational step is to verify the transferred archive checksums, then
wipe Gamma and build only the reviewed services on a supported OS.

## USB capture procedure

The recovery script is stored in the infrastructure repository as
`scripts/gamma_usb_backup.sh` and was copied to Gamma at
`~/gamma_usb_backup.sh`. It creates one timestamped ZIP directly on a mounted
USB drive, tests every ZIP entry, writes a SHA-256 checksum file, verifies that
checksum, and calls `sync` before reporting success.

It stops the Minecraft PM2 process only while copying its world and uses an
exit trap to restart it even if the ZIP or validation step fails. The script
excludes `node_modules/` from TelegramBot and Hall, and excludes TelegramBot's
`.env`. The token must be rotated and recreated on the replacement machine.

After confirming the USB mount path with `lsblk`, run on Gamma:

```bash
bash ~/gamma_usb_backup.sh /media/manuel/USB_LABEL
```

Only remove the USB after the script reports that the checksum is `OK` and the
write cache has been flushed. The resulting USB contents remain untrusted data:
scan and inspect them on the replacement machine before extracting or running
anything.
